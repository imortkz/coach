# Roadmap: GymCoach

## Overview

GymCoach delivers a personal workout tracking app across five phases following the critical dependency chain: database foundation and seed data first, then exercise and program management (define what you lift), then workout logging (the make-or-break gym screen), then history and progression intelligence (see what you did and what to lift next), and finally CLI analytics and data tools. Each phase delivers a complete, verifiable capability.

## Phase Details

### Phase 1: Foundation and Schema
**Goal**: All three repos are scaffolded with working dev environments, the SQLite database schema exists with proper plan-vs-log separation, migrations run cleanly, and the exercise library is seeded with data
**Depends on**: Nothing (first phase)
**Requirements**: SCRP-01, SCRP-02
**Success Criteria** (what must be TRUE):
  1. Running the migration script creates all database tables (exercises, programs, program_exercises, workouts, workout_sets) in SQLite with WAL mode enabled
  2. Running the seed script populates the exercise library with categorized exercises (by muscle group and equipment)
  3. The FastAPI backend starts and serves a health endpoint; the Vue frontend starts and renders a page; both connect to the same SQLite database
**Plans:** 3/3 plans complete

Plans:
- [x] 01-01-PLAN.md — Scripts repo: SQLAlchemy models, Alembic migrations, and exercise seed data
- [x] 01-02-PLAN.md — Frontend repo: Vue 3 + Vite + Tailwind scaffolding with router and stub views
- [x] 01-03-PLAN.md — Backend repo: FastAPI scaffolding with health endpoint, Dockerfiles, and integration checkpoint

### Phase 2: Exercise and Program Management
**Goal**: Users can browse the exercise library, create custom exercises, and build workout programs with per-set targets (reps, weight, warmup flag) via a responsive web UI
**Depends on**: Phase 1
**Requirements**: EXER-01, EXER-02, PROG-01, PROG-02
**Success Criteria** (what must be TRUE):
  1. User can browse exercises filtered by muscle group and equipment in the web UI
  2. User can create a custom exercise with name, muscle group, and equipment and see it appear in the library
  3. User can create a workout program by selecting exercises and setting target sets, reps, and weight for each
  4. User can edit and delete existing programs
**Plans:** 4/4 plans complete

Plans:
- [x] 02-01-PLAN.md — Schema migration (program_sets table) and Exercise CRUD API with tests
- [x] 02-02-PLAN.md — Program CRUD API with nested per-set targets and tests
- [x] 02-03-PLAN.md — Responsive navigation and Exercise Library UI with filters and inline CRUD
- [x] 02-04-PLAN.md — Program builder UI with exercise picker, per-set targets, and visual checkpoint

### Phase 3: Workout Logging
**Goal**: Users can run a workout session at the gym -- start from a program template, log each set with actual weight and reps (pre-filled from last session), correct mistakes, and track rest between sets
**Depends on**: Phase 2
**Requirements**: LOG-01, LOG-02, LOG-03, LOG-04
**Success Criteria** (what must be TRUE):
  1. User can start a workout from an existing program and see all exercises with target sets/reps/weight pre-loaded
  2. User can log a completed set with weight and reps, with values pre-filled from the last session for that exercise
  3. User can edit or delete any logged set, exercise entry, or entire workout
  4. A rest timer auto-starts after logging a set and counts down a configurable duration
**Plans:** 3/3 plans complete

Plans:
- [x] 03-01-PLAN.md — Workout API: CRUD endpoints, pre-fill logic, settings, migration for rest_timer_disabled, and integration tests
- [x] 03-02-PLAN.md — Core workout UI: program picker, active workout view, exercise cards, set rows with tap-to-complete
- [x] 03-03-PLAN.md — Rest timer, edit/delete with swipe and undo toast, workout summary, and visual checkpoint

### Phase 4: History and Progression
**Goal**: Users can review past workout data and receive intelligent weight suggestions for their next session based on logged performance
**Depends on**: Phase 3
**Requirements**: HIST-01, HIST-02, PRGS-01, PRGS-02
**Success Criteria** (what must be TRUE):
  1. User can view a chronological list of completed workouts showing exercises performed and sets logged
  2. User can view per-exercise history showing weight and reps across recent sessions
  3. During a workout, user sees auto-suggested weight for each exercise based on past performance (hit all target reps at weight X means suggest X + increment)
  4. The progression algorithm correctly identifies when a user has met their rep targets and suggests an appropriate weight increase
**Plans:** 3/3 plans complete

Plans:
- [x] 04-01-PLAN.md — Backend: history endpoints, progression algorithm, and tests
- [x] 04-02-PLAN.md — Frontend: workout history list with expandable cards and program filter
- [x] 04-03-PLAN.md — Frontend: per-exercise history page with charts and suggestion indicator in workout flow

### Phase 5: Analytics and Data Tools
**Goal**: Users can generate progress reports from the command line and export/import their data for backup
**Depends on**: Phase 4
**Requirements**: SCRP-03, SCRP-04
**Success Criteria** (what must be TRUE):
  1. User can run a CLI command that shows estimated 1RM trends, volume per muscle group, and workout frequency
  2. User can export all workout data to JSON or CSV format
  3. User can import a previously exported JSON/CSV file to restore data
**Plans:** 2/2 plans complete

Plans:
- [x] 05-01-PLAN.md — CLI analytics subcommands (1RM trends, volume by muscle group, workout frequency) with Rich tables and JSON output
- [x] 05-02-PLAN.md — Data export (JSON/CSV) and import (JSON with auto-backup and replace-all)
